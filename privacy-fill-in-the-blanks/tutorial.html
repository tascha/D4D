Privacy Fill-in-the-Blanks Story Tutorial

Welcome to this fill-in-the-blanks storytelling project. Today you'll work with a script that takes user input from a webpage and turns it into a story using helper text stored inside code.

You'll learn about

  • forms and inputs.
  • writing document statements.
  • mixing and matching strings and variables to tell a story.

Here are the steps you'll follow:

  1. What's a form?
  2. What's an input?
  3. What does document do?
  4. What does our project do?
  5. Rewrite the story
  6. Change the input labels
  7. Mess with the code
  8. Give yourself credit and publish

Step-by-Step Instructions

  1. What's a form?

    A form on a webpage is a lot like a form in real life. It's a container for a bunch of fill-in-the-blanks boxes and lines that you fill out. Think of a workseheet at school, a form at the doctor's office, or a job application. These are all examples of forms.

    The Mozilla Developer Network defines form like this:

    The HTML <form> element represents a document section that contains interactive controls to submit information to a web server.

    You can see an example of a form and the inputs it holds at line 30 of the file called index.html in the FILES sidebar to the left.

  2. What's an input?

    Each blank, box, or line in a formthat needs you to fill something in is an input.

    Working together, forms and inputs create webpages through which users can do things like buy something, complete a survey, or fill out an online application.

    A form can hold any number of inputs inside of it. It's good practice to make sure that each of your inputs goes with the form it belongs to. For example, you wouldn't create an input for "your favorite piece of clothing" in a form about favorite foods.

    The Mozilla Developer Network describes an input this way:

    The HTML element <input> is used to create interactive controls for web-based forms in order to accept data from the user. How an <input> works varies considerably depending on the value of its type attribute.

    There are many types of input. Today you'll work with text inputs, though other kinds like radial buttons and range sliders exist.

  3. What does document do?

    According to the Mozilla Developer Network,

    the Document interface represents any web page loaded in the browser and serves as an entry point into the web page's content

    What does that mean?

    Well, it means that in JavaScript, you can use document as the beginning of a path to get and maniuplate any part of a webpage (which is the document itself) connected to a script. For example, using document, we can go into a webpage, find its input elements, and get whatever a user types into them. Then we can take that user input and do things with it like make calculations or store it as data.

    In today's project, you'll generate stories with the user input you collect using document.

  4. What does our project do?

    Click on the file called index.html in the FILES sidebar to the left. Visit its preview pane (on the right) and "play" through the page a few times. That is to say, fill in the blanks and then hit the button to see how the page gathers your user input and transforms it into a story.

    Look at the code for index.html, as well. Notice that each input in the form at line 30 has an id, a unique identification. Our script uses those unique ids to gather user input from the webpage, or document.

    Click on script.js to see how. Look at the code between line 7 and line 12. Notice that the script is setting up vairables that follow a path back into the document to get the input from each box on the webpage.

    In this set-up, a line like...

    var firstWord = document.getElementById('box1').value;

    ...really means make the first variable (which is also the first piece of user input we'll use in our story) equal to whatever value a user types into the input that has box1 for its id. We are going to "get" that input by its id (which is box1) and make our variable called firstWord equal to the value of that input.

    There are six inputs to collect from our webpage, and once the script gets them, it plugs those words into a kind of fill-in-the-blanks story at line 18 of script.js.

    This time, though, instead of getting a value from the webpage, our document pathway gets the element with story-space for its id and then replaces that element's innerHTML with this story (including all the user input gather above). This time document pushes text into the webpage instead of getting a textual value from it.

    Notice that the story is a mix of strings and variables. The strings hold all the HTML and helper text for the story - they are the skeleton of the story. The strings have to appear in quotes so they can show up exactly as they're written on the webpage.

    The variables show up between strings to deliver the user input from the page.

  5. Rewrite the story

    To begin making the project your own, start revising the story told inside the strings at line 18. Be creative, but stay within the boundaries of what's acceptable for your community. Try to change only the words inside the story - do not change the HTML tags in the strings or the variables between the strings.

    Your variables should stay outside of the strings - which means outside of the quotation marks. You can add separate strings and variables to the story using the plus sign, + between different parts. You do not need a + at the end of the story.

    Try to work in all six variables, as well. You can change the kinds of words you ask for if you need to on index.html in the next step.

  6. Change the input labels

    Jump back into index.html. Visit the input elements between line 31/code> and line 36. Find the label for each input, like "singular noun" or "verb." Change those labels to ask for the parts of speech your story needs.

    For example, if your story needs an adjective, or describing word, from the input with the id box1, then delete "singular noun" and type "adjective" instead. You can go back and forth between index.html and script.js to keep track of what kinds of words you need. Be sure to ask for help if you wonder what kinds of words to use for each label.

    After you change the storytelling function on script.js and the labels on index.html, test your story a few times and identify any spots that need proof-reading and trouble-shooting. Ask a classmate or teacher for help if you get stuck. Debugging code can be tough for everyone, even coders!

  7. Mess with the code

    Remember that you can visit the file called style.css, as well, to play with the styling instructions for the page. Change the values of attributes like color or font-size to see what happens to index.html as a result. Try to break anything you fix, but don't be afraid to take some risks. You can always hit undo!

  8. Give yourself credit and publish

    Finally, go to line 43 of index.html and change the text to give yourself credit for crafting a fill-in-the-blanks story of your own.

    After you revise the credits, hit the "Publish" button in the upper right-hand corner of the screen, describe your project, and then grab its URL to share with friends and family.

Congratulations on completing your Privacy Fill-in-the-Blanks Story! What else can you do with forms, inputs, HTML, and JavaScript? Keep playing and learning to discover more ways to write the web and to express yourself creatively online!
